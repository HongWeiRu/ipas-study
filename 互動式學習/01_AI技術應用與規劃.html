AI 相關技術應用

在進入「規劃師」的職能訓練之前，必須先了解 AI 在四大子領域的核心技術：NLP（語言）、CV（影像）、生成式（創造）、多模態（融合）。這四大領域是當代 AI 應用的主軸，也是後續評估、選型、導入決策的基礎。

本章四大子主題 3.1 NLP：讓機器理解、處理、生成自然語言；從規則式 → 統計 → 深度學習 → 預訓練語言模型。
3.2 CV：讓機器「看懂」影像 — 從特徵抽取、CNN 到物件偵測與影像分割。
3.3 生成式 AI：能產出新內容（文／圖／音）的 AI；GAN、VAE、Diffusion、Transformer。
3.4 多模態 AI：同時處理多種資料型態的整合架構；CLIP、GPT-4V、Gemini。

自然語言處理（NLP）技術與應用

2022 年底 OpenAI 推出 ChatGPT，以對話式 AI 的形式迅速吸引全球矚目，推動各行業對 NLP 應用的探索。自然語言數據的價值已逐漸超越傳統結構化數據 — 企業透過 NLP 能精準分析用戶意圖、洞察市場趨勢、提升決策效率。

B · NLP 技術演進的四個階段

NLP 四世代技術演進

1980s – 1990s
①
規則式方法
Rule-based
核心思想

專家手工編寫語法規則與詞彙辭典，明確定義語言結構與語法關係。

解決的痛點

解決早期缺乏自動化、無法理解語意的問題；具高可解釋性，處理邏輯可清楚追溯。

侷限

完全依賴專家知識；擴展性差，無法處理多變語境；維護成本高。

代表技術與模型
ELIZA早期對話式 AI；以關鍵詞比對模擬對話，無語意理解。
專家系統人工規則推理；專注特定領域知識，範圍狹窄。
1990s – 2010
②
統計語言模型
Statistical LM
核心思想

以大量語料訓練機率模型，自動學習語言中的統計結構與規律。

解決的痛點

突破規則式僵化；能處理較大範圍語言變化，提高靈活性與可擴展性。

侷限

缺乏深層語意理解；依賴大量標注語料；難以捕捉長距離依賴。

代表技術與模型
N-gram計算詞彙共現機率以預測語言結構；長語境表現差。
HMM 隱馬可夫機率模型；應用於語音辨識、詞性標注。
CRF 條件隨機場圖形機率模型；命名實體辨識 (NER) 常用。
2010 – 2018
③
深度學習模型
Deep Learning
核心思想

用神經網路自動學特徵，處理語言序列；LSTM 引入記憶單元解決梯度消失。

解決的痛點

長距離依賴捕捉能力大幅提升；不再依賴人工設計特徵，可學習更深層結構。

侷限

訓練計算資源大；難以平行運算（序列遞迴）；長序列仍有效能限制。

代表技術與模型
RNN 循環神經網路處理序列數據；長序列易梯度消失。
LSTM 長短期記憶RNN 改良；解決梯度消失，翻譯/語音突破。
GRU 門控循環單元LSTM 簡化版；結構單純、運算高效。
2018 – 至今
④
預訓練語言模型
Pre-trained LM
核心思想

自注意力機制 + 全平行架構；先大規模預訓練、再下游微調 (Fine-tuning)。

解決的痛點

同解長距離依賴與多任務遷移；少量標注即可獲得高效能，大幅降低開發門檻。

侷限

運算資源需求極高；推理效率仍是挑戰；模型規模龐大。

代表技術與模型
Transformer (2017)核心：自注意力 + 全平行；長距依賴突破。
BERT首個雙向預訓練模型；MLM 遮蔽語言建模。
GPT自迴歸生成；單向預訓練，擅長對話與文本生成。
RoBERTa／T5／PaLM 為進階優化變體
每一階段的突破都是對「語言結構理解深度」的提升
補充：為什麼 Transformer（2017）會徹底改變遊戲規則？ ① 自注意力機制（Self-Attention）讓每個詞可同時考慮序列中其他所有詞，解決長距離依賴。
② 全平行架構不像 RNN 必須逐字運算，訓練速度大幅提升。
③ 可擴展至極大規模語料（GPT-3 用 45TB 訓練資料、175B 參數）。
後續 BERT（雙向理解）、GPT（自迴歸生成）、T5（文字到文字）都建立在這個基礎上。
常見疑問：四世代為什麼沒有獨立的「機器學習」階段？
因為 機器學習（ML）是上位概念，不是平行階段。第 2、3、4 階段全都是 ML，只是用不同方法。AI / ML / DL 是三層包含關係：
AI · 人工智慧 Artificial Intelligence
最廣的概念 — 讓電腦做「智慧」的事
⊃ 非 ML 的 AI：規則式系統、專家系統 ← NLP 第 ① 階段（規則式）屬於這層
ML · 機器學習 Machine Learning
AI 的子集 — 從資料學習規則，而非人工寫死
⊃ 傳統 ML：N-gram、HMM、CRF、SVM、決策樹、Naive Bayes、隨機森林… ← NLP 第 ② 階段（統計）屬於這層
DL · 深度學習 Deep Learning
ML 的子集 — 用「深層神經網路」做 ML
RNN · LSTM · GRU · Transformer · BERT · GPT… ← NLP 第 ③ ④ 階段都屬於這層
關鍵分水嶺：規則式 vs 機器學習
真正的「典範轉移」在於「規則由誰決定」：
① 規則式（非 ML） — 規則由人類專家手寫（例：「if 句子含『不』，then 是否定句」）。可解釋性高但無法泛化。
② 機器學習（ML） — 規則由演算法從資料中自動學習。可泛化但較難解釋。
所以四世代中只有「規則式 → 統計」這個跨越才是真正的「從非 ML AI 到 ML 的典範轉移」；統計 → 深度學習 → 預訓練則都是在 ML 內部的方法升級（從淺到深、從靜態到語境型、從單任務到通用預訓練）。
※ 進階追問「DL vs 傳統 ML 最大差異」與 7 個衍生差異 已移到 E · 深度學習在 NLP 中的應用 節 — 那裡是探討 DL 的天然位置。

C～F 整體脈絡：NLP 處理流水線

三條串起來的演進線（讀 D / E / F 時請對照）
① D 向量化本身也在演進：One-hot → BoW → TF-IDF → N-grams → Word2Vec → BERT，從稀疏到密集、從靜態到語境型。
② E 模型架構沿著同一條線：RNN → LSTM → GRU → Transformer，解決長距離依賴與平行運算。
③ F把 D + E 整合並加上「大規模預訓練 + 微調」這個範式 — BERT 是 D + E 的結合產物，GPT 系列則是 F 範式的代表。所以你會在 D 與 F 都看到 BERT、GPT — 它們同時是「向量產生器」也是「下游任務模型」。
回頭對照 A 節「NLP 三個核心任務層次」 — 流水線怎麼對應 理解 / 處理 / 生成？
A 節把 NLP 切成三個任務層次：理解 Understand（NLU）／處理 Process／生成 Generate（NLG）。把它疊到 C–F 流水線上：
① C 前處理 ＋ D 向量化 ＝ A 節的「處理 Process」層 — 把語言轉為電腦可操作的結構（C 做結構化、D 做數值化）。
② E 深度模型是中間的「語意處理引擎」 — 既支撐理解也支撐生成，看你用 Encoder（BERT 系，偏理解）還是 Decoder（GPT 系，偏生成）。
③ F 預訓練 + 微調是「理解 NLU + 生成 NLG」的具體應用層 — BERT 偏理解類任務（分類、QA、NER）；GPT 偏生成類任務（對話、續寫、創作）。
結論：D 詞彙向量化屬於「處理 Process」層，與 C 一起完成「語言轉電腦結構」這件事；A 節的「處理」其實是 C+D 兩節合起來。

補充：為什麼 POS 是「橋樑」？— POS → NER → 關係抽取的串聯
POS Tagging 被稱為「基礎語法與高階語意分析的橋樑」，是因為它先把每個詞的語法身份標清楚，後面更複雜的語意任務才能在此基礎上抽取資訊。實務串聯如下：

1. NER 命名實體辨識（Named Entity Recognition）
從文本中找出「具有特定意義的實體」 — 人名、地名、組織、日期、金額、產品型號等。NER 嚴重依賴 POS：
① 大部分命名實體都是名詞或專有名詞（NNP），POS 直接告訴模型「哪些 token 是候選實體」。
② 經典規則式模式 NNP was born → 標為 NNP 的詞極可能是「人名」。
③ 大小寫 (Capitalization) + POS 一起作為特徵，可大幅提升精度。

2. 關係抽取（Relation Extraction）
辨識實體之間的關聯 — 「張三 is_CEO_of A 公司」、「藥物 X treats 疾病 Y」。流程是先 NER 找實體 → 再用 POS + 句法結構判斷實體間的動詞 / 介系詞關係。動詞（VB）承載關係資訊，介系詞（IN）連接的兩個名詞片語常隱含特定關係。

3. 特徵工程（Feature Engineering）中的角色
在傳統機器學習模型（CRF、SVM）裡，POS 標籤直接作為輸入特徵欄位。即使在深度學習時代，POS 仍常作為輔助訊號強化模型對語法結構的理解 — 特別在低資源語言或專業領域文本（醫療、法律）中尤其關鍵。

4. 實務例
① 醫療文本：POS → 識別藥物名 (NNP) 與症狀 (NN) → 抽取「藥物–治療–疾病」三元組建構知識圖譜。
② 金融新聞：POS → 識別公司、職位、金額 → 抽取併購、人事異動關係。
③ 法律文書：POS → 識別當事人、案件、法院 → 自動建立案件結構化資料（對應到後面 §3.1 G 的 NER 與關係抽取應用情境）。

概念框架：兩個獨立維度

「分布式 vs 非分布式」和「語境型 vs 非語境型」常被混用，但其實是兩個獨立維度 — 一個技術會同時被這兩個維度標記。先把兩個維度拆開看：

維度一：分布式 vs 非分布式

回答：「詞向量怎麼產生？」
非分布式 — 每個詞獨立配一個固定向量，與其他詞無關（純查表）。
分布式 — 詞向量由它與其他詞的「共現關係」學出來；語意相近的詞向量自然相近。

維度二：語境型 vs 非語境型

回答：「同一個詞在不同句子裡，向量會不會變？」
非語境型 — 詞向量是查表式，"bank" 永遠是同一個向量。
語境型 — 詞向量依當下句子動態計算，"bank" 在「金融」與「河岸」句中產出不同向量。

① 查表 vs 統計：兩者都不需要學習、由規則直接產出。差別在 — 查表只給 0/1（沒有意義）；統計給出有意義的數值權重（如 TF-IDF 反映詞的重要性）。

核心原理

結合「詞在單一文件的出現頻率（TF）」與「該詞在整體語料的稀有程度（IDF）」 — 強調文件中的關鍵詞、降低常見詞權重。比 BoW 進一步，能反映詞的「重要性」而非單純次數。

三條公式
TF(t, d) = 詞 t 在文件 d 中出現次數 / 文件 d 總詞數
IDF(t, D) = log( |D| / |{d ∈ D : t ∈ d}| )
TF-IDF(t, d, D) = TF(t, d) × IDF(t, D)

t = 詞、d = 單一文件、D = 全部文件集合

計算示例

情境：語料 1,000 篇文件；詞「AI」只在 10 篇中出現；某文件總共 100 詞、含 3 次「AI」。

TF = 3 / 100 = 0.03
IDF = log(1000 / 10) = log(100) ≈ 2
TF-IDF = 0.03 × 2 = 0.06

優點 / 限制 / 應用
  • + 強調關鍵詞、降低常見詞影響
  • + 實作簡單、效果穩定
  • − 無法捕捉語意關聯
  • − 無法處理多義詞
  • 應用：搜尋引擎排序、文件檢索、簡單分類
一句話：常見詞便宜、稀有詞貴 — TF-IDF 秤出每個詞對文件的「重要性」。

  • 靜態詞向量 — bank 在「銀行」與「河岸」是同個向量
  • 無法處理多義詞 — 因為向量靜態，無法區分同詞異義
  • 低頻詞表現較差 — FastText 改進 OOV 但仍有限
  • 高維度與稀疏問題 — GloVe 尤其明顯
Word2Vec · 2013 · Google (Mikolov et al.)

基於淺層神經網路，有兩種對偶架構：CBOW（用上下文預測中心詞 — 訓練快）vs Skip-gram（用中心詞預測上下文 — 對低頻詞更有效）。架構幾乎相同，只是目標函數方向相反。

經典示例：king − man + woman ≈ queen — 展示向量可進行語意邏輯運算。
GloVe · Global Vectors · 2014 · Stanford

不依賴局部上下文（像 Word2Vec 的 Skip-gram），而是利用整體語料庫的詞共現統計（共現矩陣），透過矩陣分解（類似 SVD）學習每個詞的靜態向量表示。是「統計式」與「神經式」之間的橋樑方法。

例：「king」常與「throne、royal」共現；「queen」也常與「throne、royal」共現 → GloVe 從共現統計學到 king 與 queen 位置接近，捕捉語意對等。
FastText · 2016 · Meta (Facebook AI Research)

將每個詞切為 n-gram 子詞（subwords）學習向量（常用 3–6 字母），最後將所有子詞向量加總或平均作為整詞向量。最大貢獻：能處理 OOV（從未見過的詞）。

"playing" → "pla", "lay", "ayi", "yin", "ing"
"unhappiness" → <un, unh, hap, app, ppi, pin, ine, nes, ess>
關鍵：即使沒見過 "nationalization"，也能從 "nation"、"-al"、"-ization" 等子詞組合出語意接近 "nation" 的向量。
一句話：從詞與詞的共現關係「學」出語意向量；訓練完查表使用、向量是靜態的。
No. ⑥懂
語境型詞嵌入 — ELMo / BERT / GPTContextualized Embeddings · 當代主流 · 2018+
深度 DL
核心突破

詞的語意會根據上下文動態改變，因此詞向量也應隨語境動態生成。這突破了傳統「每個詞只有一個固定向量」的限制，能處理多義詞、語意歧義、語境轉折、反語、代詞消歧等複雜現象。建立在深度語言模型（LSTM 或 Transformer）之上，並透過大規模語料預訓練。

優點（4 項）
  • 語境感知能力強 — 解決多義詞與語意變化
  • 任務效果優異 — NER、閱讀理解、機器翻譯領先
  • 支援遷移學習 — 預訓練後微調適應各任務
  • 豐富語意建模 — 語境推理、代詞消歧
限制（3 項）
  • 硬體與資源需求高 — 需大量 GPU / TPU
  • 訓練成本高 — 預訓練資料龐大
  • 部分模型單向限制 — GPT 僅看前文，純理解任務略受限
ELMo · Embeddings from Language Models · 2018 · AllenNLP

基於雙向 LSTM（BiLSTM）語言模型。不僅考慮詞彙本身，亦將句子的語法與語意結構納入詞向量生成；可整合進現有模型增強語意表示。

例：在 "The bank approved the loan" 中，ELMo 會根據 "approved"、"loan" 等上下文，推斷 bank 指金融機構而非河岸。
BERT · Bidirectional Encoder Representations from Transformers · 2018 · Google

基於雙向 Transformer 編碼器。預訓練任務有兩個：MLM（遮蔽語言建模，遮 15% token 讓模型猜）+ NSP（句子關聯預測）。下游任務：問答、分類、NER 等理解類任務。

多義詞例：BERT 能辨別 "He sat on the river bank"（河岸）與 "She went to the bank to withdraw cash"（金融機構）產出不同向量。
GPT 系列 · Generative Pre-trained Transformer · 2018+ · OpenAI

基於單向 Transformer 解碼器（Decoder），採自迴歸生成（Autoregressive Prediction）— 一個字接一個字預測。可搭配 Prompt 進行生成任務，支援 Few-shot / Zero-shot。下游任務：對話生成、摘要、創作、翻譯。ChatGPT 即此架構。

例：給 prompt「請寫一段關於 AI 發展的介紹」→ GPT 從左到右逐字生成語法語意通順的段落，具高度語境一致性。
一句話：用大規模預訓練的深度模型「懂」整句語境 — 同個詞在不同句子產出不同向量。

E · 深度學習在 NLP 中的應用

從 D 節「向量化」往下走，下一步是把向量序列丟進什麼模型去處理。本節介紹 NLP 領域四大深度學習模型架構（RNN / LSTM / GRU / Transformer）。在介紹模型之前，先釐清「深度學習」跟「傳統機器學習」到底差在哪 — 因為這個差異決定了為什麼 NLP 從第三世代後幾乎全面轉向 DL。

深度學習 vs 傳統機器學習 — 根本差異

一句話答案：「特徵由誰決定」
傳統 ML 要人類設計特徵（特徵工程 Feature Engineering）；深度學習讓神經網路自動學特徵（特徵學習 Feature Learning）。這是唯一根本差異，其他差異都從這裡衍生。
處理流程對照 — 同樣的任務：分類「這是貓還是狗？」
傳統 ML（多階段、靠人類）
原始圖片 ▸ 人工設計特徵（耳朵形狀／毛色／鬍鬚／體型…） ▸ SVM ／ 決策樹 ／ 隨機森林 ▸ 貓 ／ 狗
特徵的好壞完全取決於人類專家，特徵設計不好模型再強也沒用
深度學習（端到端、機器自學）
原始圖片 ▸ 深層神經網路（卷積層 × N → 自動學邊緣／紋理／形狀／物件部位） ▸ 貓 ／ 狗
特徵由神經網路從原始資料自動學出，人類完全不介入特徵設計（End-to-End）
關鍵差別：傳統 ML 把「特徵設計」交給人類；深度學習把「特徵學習」也納入網路，End-to-End 自動完成。

7 個衍生差異 — 都是「特徵自動學」帶來的連鎖效應

面向傳統機器學習 (ML)深度學習 (DL)
① 特徵處理
根本差異		 人類設計（特徵工程）— 需領域知識、耗時、限制效能上限 神經網路自動學（特徵學習）— 端到端，從原始資料直接到結果
② 網路深度 淺層 / 單隱藏層（或無神經網路）— 決策樹、SVM、線性模型 深層（多隱藏層，可達數十至數百層）— CNN、RNN、Transformer
③ 資料需求 中小型資料即可表現良好（千～萬筆） 需要大量資料（百萬～數十億筆），否則容易過度擬合
④ 計算資源 CPU 通常足夠，可在筆電上跑 需要 GPU ／ TPU，訓練時間長、能耗高
⑤ 可解釋性 高 — 特別是線性模型、決策樹；可解釋每個特徵對結果的影響 低（黑箱）— 特徵由網路內部學出，難以解釋具體決策邏輯
⑥ 適合資料型態 結構化資料（表格、數值）— 金融風控、信用評分、客戶分群 非結構化資料（影像、文字、音訊、影片）— 人類難設計特徵的領域
⑦ 效能 vs 資料量關係 達到一定資料量後效能會飽和（plateau） 資料越多效能越強（scaling laws — GPT-3、PaLM 等大模型的核心觀察）
為什麼這 7 個差異全部源自「特徵自動學」？— 因果鏈
① 要自動學特徵 → 需要可學習的參數很多 → 深層網路（② 網路深度）
② 參數多 → 需要更多訓練資料才不會 overfit → 大資料需求（③ 資料需求）
③ 深層 + 大資料 → 訓練計算量爆增 → GPU / TPU 需求（④ 計算資源）
④ 特徵不是人類定義的 → 無法直接解釋 → 黑箱問題（⑤ 可解釋性）
⑤ 能自動學複雜模式 → 在人類難設計特徵的領域（影像、文字、音訊）大幅勝出（⑥ 資料型態）
⑥ 網路容量大 → 資料越多越學得到 → 越多越強（⑦ scaling laws）
所以理解「特徵由誰決定」這一條，等於同時理解了 ML vs DL 的全部差異。
常考點：什麼時候選 ML、什麼時候選 DL？
  • 結構化資料（表格、數值）+ 資料量不大（< 數萬筆）→ 選傳統 ML（XGBoost、LightGBM 常贏）
  • 非結構化資料（影像、語音、文字）+ 資料量大 → 選深度學習
  • 需要解釋每個決策依據（醫療、金融風控、法律審查）→ 選傳統 ML（或 ML + XAI 工具）
  • 有預訓練模型可用（NLP / CV 大多如此）→ DL 微調往往最快達成目標，不需從零訓練
  • 邊緣裝置部署（手機、IoT）→ ML 或小型 DL 模型（LoRA / 蒸餾 / 量化）

NLP 中的四大深度學習模型架構

釐清 DL vs ML 的根本差異後，回到 NLP — 處理「語言序列」這種特殊資料型態，深度學習圈發展出以下四大主流架構。延續 D 節的動詞記憶法，給每個模型一個關鍵動作：循 (RNN) → 記 (LSTM) → 簡 (GRU) → 注 (Transformer)，沿著「解決長距離依賴 + 平行運算」這條主線演進。

4 個動作記住 4 個模型 + 演進邏輯
① 循 RNN：循環處理時間步 → 但會梯度消失，記不住遠處資訊
② 記 LSTM：引入「記憶單元 + 三閘」選擇性記憶 → 解決梯度消失，但結構複雜
③ 簡 GRU：把 LSTM 的閘簡化（2 閘取代 3 閘）→ 訓練快、效果接近
④ 注 Transformer：丟掉循環、改用自注意力同時看所有詞 → 解決長距依賴 + 全平行訓練（這條路通往 BERT/GPT，也就是 F 節主題）
No. ①循
RNN — 循環神經網路Recurrent Neural Network · 1980s+ · 序列處理的起點
基礎 DL
核心機制

每個時間步的隱藏狀態 hₜ 同時依賴「當前輸入 xₜ」與「前一步隱藏狀態 hₜ₋₁」，形成循環（recurrent）。這讓網路具備「記憶」概念 — 之前讀到的內容會影響現在的處理。

優勢
  • 能處理時間序列與語序
  • 結構簡單，是所有序列模型的祖先
限制
  • 梯度消失/爆炸嚴重 — 長序列時遠處資訊傳不過來
  • 訓練不穩定、無法平行（必須逐步）
一句話：把時間「循環」進神經網路，但長序列就忘了。
No. ②記
LSTM — 長短期記憶Long Short-Term Memory · 1997 · 解決 RNN 梯度消失
改良 RNN
核心機制：三閘門 + 細胞狀態（Cell State）

在 RNN 基礎上新增細胞狀態 Cₜ（像一條「記憶高速公路」）+ 三個閘門來控制資訊流：

三閘各自的角色
  • 遺忘閘（Forget）：決定丟掉 Cₜ₋₁ 中哪些舊資訊
  • 輸入閘（Input）：決定加入哪些新資訊到 Cₜ
  • 輸出閘（Output）：決定輸出哪些 Cₜ 內容到 hₜ
優勢 / 限制 / 應用
  • + 能選擇性記憶，捕捉長距依賴
  • + 解決 RNN 梯度消失問題
  • − 結構複雜、訓練耗時
  • − 仍需逐步運算，無法平行
  • 應用：機器翻譯、語音辨識、ELMo 詞向量
一句話：用「細胞狀態 + 三閘」選擇性記住遠處資訊。
No. ③簡
GRU — 門控循環單元Gated Recurrent Unit · 2014 · LSTM 的精簡版
改良 LSTM
核心機制：2 閘取代 3 閘

把 LSTM 的「遺忘閘 + 輸入閘」合併成更新閘（Update Gate）；新增重置閘（Reset Gate）決定要忘掉多少過去資訊。沒有獨立的細胞狀態 — 直接用隱藏狀態 hₜ 同時當記憶與輸出。

GRU 兩閘：
① 更新閘 Γᵤ (Update)：決定保留多少舊資訊 + 加入多少新資訊（合併 LSTM 的遺忘 + 輸入）
② 重置閘 Γᵣ (Reset)：決定計算新候選時要忽略多少過去

與 LSTM 比較
  • 參數少 ~25%（2 閘 vs 3 閘 + 細胞狀態）
  • 訓練快，小資料集表現甚至更好
  • 長序列、複雜任務 LSTM 仍略勝
優勢 / 限制 / 應用
  • + 結構簡單、運算高效
  • + 效果接近 LSTM，資源吃緊時首選
  • − 對極長序列仍有限制
  • 應用：語言模型、機器翻譯、語音合成
一句話：LSTM 的「簡化版」 — 2 閘代 3 閘，訓練更快、效果接近。
No. ④注
Transformer"Attention is All You Need" · 2017 · Google · 當代 NLP 主流架構
DL 典範轉移
核心突破

2017 年由 Google 發表的論文「Attention is All You Need」徹底改變 NLP 技術版圖。丟掉「循環」結構，改用自注意力機制（Self-Attention）同時看序列中所有詞 — 一次解決長距離依賴與平行運算兩個 RNN 系列的痛點。後續所有大型語言模型（BERT、GPT、T5、PaLM、LLaMA…）都建立在這個架構上。

Transformer 三大核心概念

① Self-Attention 自注意力機制
每個詞同時學習與序列中所有其他詞的關聯權重，根據關聯程度加權表示。QKV 三向量（Query / Key / Value）是計算核心。

② Encoder–Decoder 編解碼架構
Encoder 把輸入序列轉抽象語意向量；Decoder 接收向量逐步產生目標。適合機器翻譯、摘要、Seq2Seq 任務。

③ Fully Parallel 全平行架構
不像 RNN 必須逐步處理時間步，Self-Attention 可一次處理整個序列，所有詞之間的注意力分數可並列計算。訓練速度大幅提升、適合 GPU/TPU、可擴展至 GPT-3（175B 參數）、PaLM（540B）等超大模型。

優勢
  • 長距離依賴一步搞定（不需逐字傳遞）
  • 全平行訓練，速度遠快於 RNN 系
  • 可擴展至超大規模（百億～兆參數）
  • BERT、GPT、T5 等預訓練模型的共同基礎
限制
  • 運算資源需求極高（GPU/TPU 必備）
  • 自注意力複雜度 O(n²) — 長序列吃記憶體
  • 需要大量訓練資料才能發揮
一句話：用「注意力」取代「循環」 — 同時看所有詞、全平行訓練、可擴展至超大模型。
承接 F 節：Transformer → 預訓練語言模型範式
看完 Transformer 後，下一步問題自然是：「有了這麼強的架構，怎麼最有效地使用？」答案就是 F 節要展開的「預訓練 + 微調 (Pre-train and Fine-tune)」範式 — 先用大規模語料訓練通用語言能力（預訓練），再針對特定任務微調。BERT、GPT、T5、PaLM、LLaMA 都是這個範式下的代表模型，全部建立在 Transformer 之上。

F · 預訓練語言模型與進階應用

承接 E 節 — 有了 Transformer 這個強大架構，下一步是怎麼最有效地使用它。答案是「預訓練 + 微調（Pre-train and Fine-tune）」範式：先用海量無標註語料讓模型學「通用語言能力」（預訓練），再用少量標註資料針對特定任務調整（微調）。這個範式 2018 年由 BERT 確立，至今仍是 NLP 的主流。

預訓練 + 微調 — 兩階段工作流

為什麼這個範式能稱霸 NLP？— 兩階段 + 一次學、多次用
關鍵紅利：① 預訓練只做一次但通用 → ② 微調便宜、可多次重複用於不同任務 → ③ 少量標註資料就能達到 SOTA 效能
三個關鍵紅利（為什麼這範式能稱霸）
① 樣本效率高 — 標註資料只需 1/100 ~ 1/1000 即可達相同效能（傳統從零訓練要幾十萬筆）
② 遷移學習 — 同一個 PLM 可微調到分類、QA、NER、摘要等任何下游任務
③ 降低門檻 — 大公司負擔預訓練成本（千萬美元級）；開發者只需微調（幾百美元 ~ 數千美元）
F 節三階段故事線 — 範式確立後的演進邏輯
F 節剩下的內容不是三個獨立主題，而是2018 → 2024 一條時間軸上的三個階段：
Phase 1（2018+）：預訓練範式確立 → 你該選哪個 PLM？答案在「架構」
Phase 2（2020+）：模型越來越大 → 全參數微調撐不住 → 演進到 Prompt（In-Context Learning）
Phase 3（2022+）：Prompt 也有極限 → 知識過時用 RAG、客製微調貴用 LoRA

2019		EncoderMetaBERT 變體：移除 NSP、更大批次、更多資料、訓練更久更強的理解任務基線；證明「BERT 沒訓練充分」
T5
2019		Enc–DecGoogleText-to-Text 統一框架（一切任務都改寫成「輸入文本 → 輸出文本」）多任務通用 — 翻譯、摘要、QA、分類等可同一個模型處理
GPT 系列
2018+		DecoderOpenAI單向 Transformer Decoder；自迴歸（從左到右一字字預測）生成任務主流 — 對話 (ChatGPT)、創作、續寫、Code
PaLM / LLaMA
2022+		DecoderGoogle / Meta更大規模 Transformer，數百億～兆級參數；多模態支援當代主流大模型 — 推理、Few-shot、多模態、Agent
常考點：根據任務類型選模型家族
  • 需要「深度理解文本」（分類、QA、NER、情感）→ BERT 家族（Encoder-only）
  • 需要「生成新文本」（對話、創作、續寫、Code）→ GPT 家族（Decoder-only）
  • 需要「文本轉文本」（翻譯、摘要、改寫）→ T5 家族（Encoder-Decoder）
  • 需要「同一模型處理多種任務」→ T5 或 大型 LLM（GPT/PaLM）走 Prompt 路線（見 Phase 2）
②
PHASE 2 · 2020+ · 模型規模爆炸
演進到 Prompt — 為什麼大家不再 Fine-tune 了？
核心問題：模型越來越大（百億～兆級參數），傳統全參微調撐不住。怎麼辦？
關鍵轉折：GPT-3（2020, 175B 參數）改變了遊戲規則
2018 BERT 範式：每個任務都要 fine-tune 一份模型，需要標註資料。
2020 GPT-3 論文揭示一個驚人事實：模型夠大時，連 fine-tune 都省了 — 只要「給好的 prompt + 幾個範例」就能完成新任務。這就是 In-Context Learning（上下文學習），模型不更新參數，純粹從 prompt 上下文「學」會任務。
兩個範式並排 — 為什麼 Prompt 接管了？
關鍵差別：傳統 = N 個任務需要 N 個微調模型；新範式 = 1 個 LLM + N 個 prompt 就搞定。模型越大、Prompt 越好用。
Prompt 的三層階梯 — 由易到難（依範例數量）
階梯①
Zero-Shot Prompting零範例 · 純靠任務描述 · 完全不給示範
最簡單
做法

完全不給範例，只描述任務本身，靠模型預訓練時學到的「先驗知識」推理。模型必須認得任務名詞（例：「情感分析」、「翻譯」、「摘要」）。

範例 Prompt

請判斷以下評論的情感（正面/負面/中性）：
「這家餐廳的義大利麵超好吃！」
情感：

模型輸出：正面

適用：任務簡單明確、模型對該任務有基本認知 — 大型 LLM (GPT-4/Claude) 在常見任務 Zero-shot 已經很強。
階梯②
Few-Shot Prompting給幾個範例 · 讓模型「看一下示範」就學會
最常用
做法

在 prompt 中提供 3–10 個輸入/輸出範例，讓模型從這些示範中學會任務模式（這就是 In-Context Learning）。GPT-3 論文核心發現：給範例的效果遠優於 Zero-shot。

範例 Prompt

範例 1：「服務很差」→ 負面
範例 2：「環境舒適」→ 正面
範例 3：「就那樣」→ 中性
---
評論：「義大利麵超好吃！」→

模型輸出：正面（學到輸出格式與標籤詞）

適用：需要特定輸出格式、模型對任務不熟、稀缺資料情境 — 5–10 個範例即可達 fine-tune 等級效能。
階梯③
Chain-of-Thought (CoT)給範例 + 推理步驟 · 教模型「想清楚再回答」
複雜推理
做法

在 Few-shot 範例中展示完整推理過程，誘導模型也用「step by step」方式思考。對數學、邏輯、多步推理類任務效果顯著提升。是 Few-shot 的進階變體（補充自 PDF）。

範例 Prompt

題目：小明有 5 顆糖，吃了 2 顆，又買了 4 顆。
推理：5 − 2 = 3，3 + 4 = 7
答：7 顆
---
題目：小華有 8 顆，給了 3 顆，再買 6 顆。
推理：

模型輸出：8 − 3 = 5，5 + 6 = 11，答：11 顆

適用：數學、邏輯推理、需多步思考的任務 — CoT 大幅提升正確率。也可只用 "Let's think step by step" 觸發 Zero-shot CoT。
③
PHASE 3 · 2022+ · 實務應用挑戰
大模型實務的兩個挑戰 — RAG 補知識、LoRA 省微調
核心問題：Prompt 雖好，但實務上遇到兩個現實問題 — ① 模型知識會過時、會幻覺；② 想客製化還是太貴。
❶ 挑戰一：知識邊界 + 幻覺

大模型只「知道」訓練時看到的資料，遇到訓練後才發生的事（如 2024 年新聞）或專業領域文件（如公司內部文件），就會：① 答不出來、② 編造看似合理但錯誤的內容（幻覺 Hallucination）。
→ 解法：RAG 給模型「外掛知識庫」

❷ 挑戰二：客製微調的代價

想讓 LLM 學特定風格、特定領域知識，傳統全參數微調 175B 模型需要：① 數十顆 A100 GPU、② 數百萬美元電費、③ 整套模型存幾百 GB。中小團隊不可能。
→ 解法：LoRA 只訓練 0.1% 參數，效果幾乎一樣

解法①
RAG · 檢索增強生成Retrieval-Augmented Generation · 給模型外掛知識庫
解決幻覺
核心思路

不去改模型參數，而是在每次提問時，先從外部知識庫檢索相關文件，把文件當作 prompt 的一部分餵給模型。模型就有了「即時最新的參考資料」可以引用 — 知識在外部、模型只負責組織與生成。

RAG 五步驟工作流
解決了什麼
  • 幻覺問題 — 答案有真實文件依據可核查
  • 知識截止日問題 — 知識庫可即時更新
  • 領域客製 — 換知識庫即換領域，不需重訓
典型應用
  • 企業內部 QA bot — 知識庫 = 內部文件
  • 醫療診斷輔助 — 知識庫 = 最新文獻
  • 客服系統 — 知識庫 = 產品手冊 + FAQ
  • 法律檢索 — 知識庫 = 判例庫
一句話：不訓練、只外掛 — 給 LLM 一個「會即時更新的外部記憶庫」。
解法②
LoRA · 低秩自適應Low-Rank Adaptation · 高效參數微調（PEFT）
解決微調貴
核心思路

傳統微調要更新模型中所有的權重矩陣 W（例如 175B 參數）。LoRA 的洞見：微調帶來的權重變化 ΔW 其實「低秩」（low-rank，可被兩個小矩陣 B·A 近似）。所以把原本的 W 凍結住，只訓練很小的 B 和 A，就能達到接近全參微調的效果。

LoRA 概念 — 把巨大的 ΔW 分解為兩個瘦長小矩陣
解決了什麼
  • 訓練成本 — 只訓 0.1% 參數，記憶體與計算需求大降
  • 儲存成本 — LoRA 模組通常 < 100MB，可保留多份切換
  • 多任務切換 — 一份原模型 + N 份 LoRA = N 個專家
典型應用
  • 邊緣裝置部署 — 手機上跑 LLaMA + 任務 LoRA
  • 多客戶 SaaS — 共用 base 模型 + 各客戶 LoRA
  • 領域客製 — 醫療 LoRA / 法律 LoRA / 程式碼 LoRA
  • 個人風格訓練 — 學使用者寫作風格
一句話：凍結大模型 + 訓練輕量小模組 — 效果 ≈ 全參微調，成本只要 1/100。
實務小結：RAG + LoRA + Fine-tune 不是互斥的，可疊加使用
這三者解決不同問題，常一起用：
① RAG：給「外部、即時更新的事實知識」（如最新財報、文件）
② LoRA：教模型「特定領域的風格與表達」（如醫療術語、法律語體）
③ Fine-tune：當資料量大、需要徹底改變模型行為時才用（最重）
典型組合：基礎大模型 + LoRA（領域風格）+ RAG（即時知識）= 一套完整的企業 LLM 應用。

G · NLP 任務與應用情境

看完前面 6 節的技術原理，現在回到實際問題：NLP 到底解決什麼問題？這些技術怎麼變成你每天用到的產品？ 本節用三大類別 + 7 個任務 + 1 個端到端案例串起來，每個任務都附真實產業範例與「背後用哪個 D-F 節學過的技術」。

三大類別 = 對應 F 節三大模型家族
NLP 任務可依「輸入輸出的本質關係」分成三類，剛好對應 F 節介紹的三大 Transformer 家族：
A 類 · 理解判斷型（文本 → 標籤/結構）→ 用 BERT 家族（Encoder-only）
B 類 · 生成轉換型（文本 → 新文本）→ 用 T5 家族（Encoder-Decoder）
C 類 · 互動對話型（多輪文本 ↔ 文本）→ 用 GPT 家族（Decoder-only）
A
CATEGORY A · UNDERSTANDING
理解判斷型 — 從文本萃取「結構化資訊」或「標籤」
輸入是自由文本，輸出是離散標籤、結構化欄位、實體關係。模型主力：BERT / RoBERTa 系列。
A.1分
文字分類Text Classification · 整段文字 → 一個類別標籤
BERT 家族
本質

給一段文本，輸出一個預定義類別。是 NLP 最基礎的監督學習任務 — 文本進去、標籤出來。

具體 I/O 範例

輸入：「我的 iPhone 突然無法充電了，需要協助」
輸出：技術支援（從 {訂單查詢, 退換貨, 技術支援, 一般諮詢} 中選）

真實產業應用
  • Gmail 垃圾郵件偵測 — Google 用 BERT 變體分類，每天處理上百億封信
  • 客服自動分派 — 中華電信、玉山銀客服系統把使用者問題路由給正確部門
  • 新聞分類 — Yahoo 新聞自動標記政治/體育/科技
  • 內容審核 — Facebook、TikTok 偵測仇恨言論
實作關鍵與挑戰

標準作法：取 BERT [CLS] token 的向量 → 接一層 Dense 分類器 → fine-tune。挑戰：① 類別不平衡（垃圾郵件 5% vs 正常 95%）；② 領域漂移（訓練時的客服語言 ≠ 上線後的真實對話）。

一句話：NLP 監督學習的「Hello World」 — 文本進、標籤出。
A.2情
情感分析Sentiment Analysis · 文本帶有什麼情緒
BERT 家族
本質

文字分類的特殊版 — 類別固定為情感標籤（正/負/中性，或更細的喜怒哀樂）。進階版叫「面向情感分析（Aspect-Based）」— 一句話對不同面向給不同情感。

I/O 範例（含面向情感）

輸入：「這支手機電池超耐用，但相機普通」
輸出（整體）：中性
輸出（面向）：電池=正面、相機=中性

真實產業應用
  • 蝦皮 / momo 評論分析 — 抽出商品的優缺點面向，給賣家改善建議
  • 品牌聲量監控 — Meltwater / Brand24 即時掃描社群
  • 金融情緒交易 — Bloomberg 把新聞情感量化，影響股價預測
  • 選舉輿情 — 政治候選人即時掌握網路風向
實作關鍵與挑戰

標準作法：BERT + 分類頭，與文字分類技術一樣。挑戰：① 反語/諷刺（「這服務真的好棒^_^」實際是負評）；② 領域用語（股市的「綠」是跌、生活的「綠」是好）；③ 表情符號與網路用語處理。

一句話：文字分類的「情緒專版」，挑戰在於人類語言的反語與隱喻。
A.3標
NER · 命名實體辨識Named Entity Recognition · 在文本中標記出「有意義的實體」
BERT 家族
本質

逐字標籤的分類任務（Token Classification） — 對每個 token 標一個類型：人名 (PER)、地點 (LOC)、組織 (ORG)、日期 (DATE)、金額 (MONEY)、產品 (PROD) 等。是從非結構化文本萃取結構化欄位的第一步。

I/O 範例

輸入：「蘋果公司執行長庫克於2024 年 9 月在加州發表iPhone 16」

輸出：
蘋果公司 → ORG · 庫克 → PER
2024 年 9 月 → DATE · 加州 → LOC
iPhone 16 → PRODUCT

真實產業應用
  • 醫療文本 — 自動標注藥物 / 症狀 / 疾病，協助醫師快速閱讀病歷
  • 金融新聞 — 抽公司、職位、金額做監管告知與預警
  • 法律文書 — 標當事人、案件、法院、條文
  • 履歷自動解析 — LinkedIn、104 用 NER 抽出技能、公司、學歷
實作關鍵與挑戰

標準作法：BERT 每個 token 的輸出向量 → 接 Token Classification 頭（通常 BIO 標記：B-開頭 / I-中間 / O-非實體）。挑戰：① 嵌套實體（「台積電供應鏈」中的「台積電」）；② 新興實體（新公司、新藥）；③ 跨語言一致性（同個實體中英文不同寫法）。

一句話：把自由文本變成結構化資料的第一步。沒有 NER，就沒有知識圖譜。
A.4連
關係抽取Relation Extraction · 找出實體之間的關係
BERT 家族
本質

NER 的下一步 — 在 NER 找出實體後，判斷實體之間的關係，輸出「三元組」(主體, 關係, 客體)。三元組是知識圖譜（Knowledge Graph）的基本單位 — 拼起來就成為機器可推理的知識網。

完整流程：原始文本 → NER → 關係抽取 → 知識圖譜
真實產業應用
  • Google Knowledge Graph — 你 Google「賈伯斯」會顯示「妻子、創辦的公司、出生地」
  • Crunchbase 企業情報 — 公司-投資人-創辦人關係網
  • 新藥研發 — 從醫學文獻抽「藥物-蛋白質-疾病」三元組
  • 金融反洗錢 — 自動建立人-公司-帳戶關係網偵測異常
實作關鍵與挑戰
  • 標準作法：先 NER 抽實體 → 用 BERT 對「兩個實體 + 周圍語境」分類關係類型
  • 挑戰 1：跨句子關係（「他」指誰要看前一句）
  • 挑戰 2：關係類型開放（醫學文獻新關係層出不窮）
  • 趨勢：用 LLM (GPT-4) 直接抽三元組已成為新主流
一句話：把零散的實體「連」起來 — 是 AI 從「閱讀」走到「理解世界」的關鍵任務。
B
CATEGORY B · GENERATION & TRANSFORMATION
生成轉換型 — 從一段文本產出「另一段文本」
輸入是文本，輸出也是文本，但內容、語言或長度都不同。模型主力：T5 / BART / GPT。
B.1譯
機器翻譯Machine Translation · 跨語言文本轉換
T5 / Encoder-Decoder
本質 & I/O 範例

典型的 Seq2Seq 任務 — Encoder 讀來源語言，Decoder 產出目標語言。

EN → ZH：
"The product launches next Tuesday"
→ 產品將於下週二發表

真實產業應用
  • Google Translate — 支援 130+ 種語言，每天上億次查詢
  • DeepL — 專業翻譯品質業界第一，歐盟法律文件常用
  • 跨境電商 — 蝦皮跨境、Amazon 自動翻譯商品描述
  • 會議即時口譯 — Zoom、Teams 即時字幕
實作關鍵與挑戰

標準作法：大規模平行語料訓練 Transformer Encoder-Decoder（早期）→ 大型多語言 LLM（當代）。挑戰：① 低資源語言（少數民族語料少）；② 文化專有詞（台灣「便當」vs 中國「盒飯」）；③ 專業術語（法律、醫療需特化模型）；④ 語法差異（中文無時態、語序差異）。

一句話：NLP 領域最早的「殺手級應用」 — Seq2Seq 架構就是為翻譯而生。
B.2寫
文字生成與摘要Text Generation & Summarization · 自動產出新文本
GPT / T5
本質

分兩大子任務：
① 摘要（Summarization） — 長文本壓縮成短版，保留關鍵資訊。又分「抽取式」（直接挑句子）與「生成式」（重新撰寫）。
② 自由生成（Generation） — 從 prompt 或主題產出全新內容（廣告、報告、創作）。

摘要 I/O 範例

輸入（1,200 字財報新聞）：「台積電今日公布第三季財報，營收達 7,597 億元⋯（略）」
輸出（100 字摘要）：台積電 Q3 營收 7,597 億，年增 39%。主要受惠 AI 晶片需求⋯

真實產業應用
  • 新聞摘要 — Bloomberg / Reuters 自動產出股市快訊
  • 廣告 / 標題生成 — Jasper.ai、Copy.ai、ChatGPT 撰寫文案
  • 程式碼註解 / 文件 — GitHub Copilot 自動產 docstring
  • 會議記錄 — Notion AI、Otter.ai 把錄音轉摘要
  • 財報重點 — 投資網站自動產業績摘要
實作關鍵與挑戰

標準作法：生成式摘要用 BART、T5、GPT；自由生成用 GPT 系列搭配 Prompt。挑戰：① 事實準確性（幻覺 — 編造不存在的數字或事件，用 RAG 解）；② 保留關鍵資訊（摘要不能漏重點）；③ 風格控制（正式 vs 口語、品牌語氣）；④ 版權問題（生成內容可能與訓練資料高度相似）。

一句話：把人類「寫」的工作交給 AI — 是 LLM 時代成長最快的應用類別。
C
CATEGORY C · CONVERSATION

綜合案例 · 智慧客服系統如何串聯多個 NLP 任務

前面 7 個任務看起來各自獨立，但真實的 NLP 系統幾乎都是多任務組合。以下用「電商客服 chatbot」展示一個使用者問題如何串連 D-G 節學過的所有技術：

使用者一句話 → 7 個 NLP 任務協作 → 完成查詢
真實的 chatbot ≠ 單一模型，而是「前處理 + 意圖分類 + NER + 狀態追蹤 + 對話生成 + RAG + 摘要」七個任務協作的管線（pipeline）
從這個案例看 D-G 節技術如何串起來
① C 節（前處理）：分詞清洗 — 任何 NLP 都從這裡開始
② D 節（向量化）：每個 token 用 BERT 變成向量 — 是 ②③ 兩步的基礎
③ E 節（深度模型）：Transformer 處理向量 — 跑在 ②③⑤⑦ 每個模型內部
④ F 節（預訓練 + 微調 / Prompt / RAG）：② 用微調過的 BERT、⑤⑦ 用 prompt 過的 GPT、⑥ 用 RAG 整合內部資料
⑤ G 節（任務）：把上述能力組裝成解決真實問題的系統
這就是為什麼這 7 節要按順序學 — 每一節都是下一節的零件。
常考點：辨識任務類型與選對模型 題目給情境問「下列哪個是最合適的 NLP 模型」 — 記住：
  • 理解類任務（分類、NER、QA、情感）→ BERT 系列（Encoder-only）
  • 生成類任務（對話、續寫、創作）→ GPT 系列（Decoder-only）
  • 翻譯 / 摘要（文本到文本）→ T5 / BART（Encoder-Decoder）
  • 多義詞處理 → 一定要用語境型嵌入（BERT/ELMo）而非靜態（Word2Vec）
  • 稀缺資料場景 → Few-shot / Zero-shot prompting，或 LoRA 微調
  • 需要最新外部知識（公司內部文件、即時資訊）→ RAG 而非只靠模型本身
  • 需要把多個任務組合（例如客服系統）→ 走 pipeline 設計，每段選對的模型

H · NLP 技術挑戰與風險

NLP 系統處理的是高度敏感與複雜的人類語言 — 客服對話、社群留言、企業內部文件、個資紀錄。當系統規模化部署，一個小偏見就可能影響百萬人。底下幾個是真實發生過的事件，沒有一個是「程式 bug」 — 都是 NLP 系統的「結構性風險」：

真實案例：NLP / AI 失控事件年表
  • 2015 Google Photos — 影像辨識把黑人照片標成「Gorillas」，Google 公開道歉並暫時移除標籤類別
  • 2016 Microsoft Tay — Twitter chatbot 上線 24 小時內被網友訓練成種族主義者，緊急下架
  • 2016 COMPAS — 美國司法用的再犯預測模型對黑人被告系統性誤判（誤判率約 2 倍）
  • 2018 Amazon Recruiting — 內部履歷篩選工具歧視女性求職者（看到 "women's" 就降分），上線後被棄用
  • 2019 Apple Card — 信用額度發生性別歧視，女性配偶得到的額度比丈夫低 10–20 倍
  • 2023 ChatGPT 律師案 — 紐約律師用 ChatGPT 寫狀紙，模型編造了不存在的判例（模型幻覺），被法官重罰
這些事件背後的風險分佈在系統的不同階段 — 有的源自資料本身、有的在訓練時學進、有的上線後才爆發。本節依「發生位置」把風險分成三層，配對應的防範方法。

①.A隱

風險情境

客服對話、郵件、社群留言、CRM 紀錄都可能含真實姓名、住址、帳號、信用卡末四碼。一旦未匿名化就餵進模型訓練，可能：① 違反《個資法》、GDPR；② 模型「記住」個資後，特定 prompt 能誘導模型「吐」出來（成員推論攻擊 Membership Inference）。

真實案例：2023 年 Samsung 工程師把機密原始碼貼進 ChatGPT 求 debug，等於把公司商業機密上傳到 OpenAI 訓練池，Samsung 隨即禁用 ChatGPT。

處理機制
  • 匿名化（Anonymization） — 徹底移除可辨識欄位，不可還原
  • 去識別化（De-identification） — 遮蔽或替換，可由代碼還原（研究/測試用）
  • 全程合規 — 蒐集、儲存、處理都遵循 GDPR / 個資法
  • Differential Privacy — 訓練時加噪音，理論保證個資不外洩（Apple、Google 已用）
必考辨析：去識別化 vs 匿名化（PDF 特別拆開）
去識別化 De-identification（可還原）

資料中可直接辨識個體的資訊被「遮蔽或替換」，但仍保有與原始個體連結的潛在可能。適用於研究、系統測試 — 可透過對應代碼重新辨識。

常見方法：
① 代碼替換（Pseudonymization）— 0912-345-678 → phone_XXYYZZ
② 資料抽象化 — 1986/07/12 → 1980 年代出生
③ 資料遮罩 — k****@example.com、王 X 明

匿名化 Anonymization（不可還原）

資料中所有可辨識個人的資訊被「徹底移除或破壞」，處理後無法再與任何特定個體連結。適用於開放資料、風險敏感環境、法規要求「無可辨識性」之處。

常見方法：
① 徹底刪除辨識欄位
② 資料隨機化 — 25 歲 → 隨機年齡分布值
③ 聚合處理 — 某區共有 500 人為 30-40 歲

一句話：「去識別化」可救回、「匿名化」一去不返 — 選哪個依「用途」與「法規等級」決定。
①.B偏
標註與品質風險Data Bias · 資料代表性不足、標註不一致
偏見源頭
風險情境

① 標註錯誤/不一致 — 不同標註員對「正面情感」定義不同，模型學到混亂訊號。
② 樣本偏斜 — 訓練資料 90% 英文 + 10% 中文 → 中文表現必然差。
③ 資料偏見 Data Bias — 語料本身含刻板印象（如「護士都是女性」、「工程師都是男性」），模型會學進並放大。

真實案例：Amazon 履歷系統用 10 年內部資料訓練，因歷史上科技業男性多 → 模型把「women's」字眼當負面訊號 → 系統性歧視女性。

處理機制
  • 標註驗證 — 用 Inter-Annotator Agreement (IAA) 量化一致性
  • 多重標註 — 同一筆資料至少 3 人標注，少數服從多數
  • 數據增強（Data Augmentation）— 反向翻譯、同義詞替換補弱勢族群樣本
  • 代表性審查 — 確保語料覆蓋不同性別/族群/方言
一句話：模型只能學它「看過」的資料 — 看不到的族群，模型就不存在。
①.C權
資料授權與版權風險Copyright · 訓練資料的法律灰色地帶
法律風險
風險情境

生成式模型訓練需要海量文本，這些文本從哪來？常見高風險來源：

  • ① 爬蟲未獲授權抓取網站內容
  • ② 公共資料但未遵循使用條款（如 CC-BY-NC 不可商用）
  • ③ 企業內部資料未經員工/客戶授權
  • ④ 自動摘要包含受版權保護段落
  • ⑤ GPT 生成商業文件與受版權內容高度相似

真實案例：2023 紐約時報控告 OpenAI 與 Microsoft，指 GPT-4 用其數百萬篇報導訓練，並能「逐字複述」其文章 — 索賠數十億美元，至今仍訴訟中。

處理機制
  • 所有訓練資料來源有可追溯授權
  • 清理階段去除未授權／受版權內容
  • 改用公開或已授權資料集（Common Crawl 加篩選、LAION 等）
  • 生成內容過抄襲檢測（Turnitin、Copyscape）
  • 公開／商用時加上版權聲明與使用條款
  • 歐盟 AI Act 要求基礎模型披露訓練資料來源
一句話：免費 ≠ 可商用 — 訓練資料的「合法性」是當代 AI 公司最大的法律風險。

②.A學
模型偏見 (Model Bias)從資料學進的偏見，會被模型「結構化」固定
公平性
本質：偏見如何「進入」模型

訓練語料若帶有文化、性別、種族偏見，模型會把這些偏見編碼進詞向量。例如 Word2Vec 在 Google News 語料上訓練後，會學到 man − doctor + woman ≈ nurse 這種充滿刻板印象的向量關係。

真實案例：履歷分類模型把男性簡歷標「管理職潛力高」、女性標「行政適合」— 因訓練語料中「男性管理階層」案例多 10 倍。
Chatbot 對男性主管描述用「果斷」、對女性用「情緒化」— 性別刻板印象被機器系統化。

處理機制
  • WEAT（Word Embedding Association Test）— 量化檢測詞向量中的隱含偏見
  • Debiasing — 移除性別/族群維度，例如 Hard-Debiasing 演算法
  • 訓練資料平衡調整 — 補弱勢族群樣本、降強勢族群權重
  • 移除模型輸入中的敏感屬性欄位（性別、種族、年齡）
  • Counterfactual 評估 — 把「他」換成「她」看模型輸出有無不同
一句話：偏見不是 bug，是模型忠實反映了世界 — 但「忠實反映歷史」不等於「對未來公平」。
②.B評

許多開源框架早就有 WEAT、SHAP、AIF360 等公平性檢測工具，但實務上：① 開發者未納入流程；② 即使檢出偏見，也缺乏量化的修正目標；③ 沒有持續監控機制 → 偏見越擴越大而無人發現。

真實案例：多語言系統對少數語言（泰文、原住民族語）的分類準確率比英文低 30–50%，但 KPI 只看「整體準確率」→ 少數族群的問題長期被掩蓋。

處理機制
  • 專案設計階段即納入公平性指標（如 Equal Opportunity、Demographic Parity）
  • 分群效能評估 — 按性別/年齡/族群分別跑指標，不只看平均
  • 多維度資料審查（不同族群、語言、年齡層）
  • 模型結果有人工覆核機制（Human-in-the-Loop）
  • 建立持續監控 dashboard 追蹤公平性指標
一句話：「沒測 = 沒問題」是錯覺 — 量化指標是公平 AI 的唯一證據。

真實案例：FAQ Chatbot 上線後新產品上市，未及時再訓練 → 持續用舊知識回答 → 使用者抱怨「機器人很笨」→ 品牌信任受損。

  • 持續再訓練 — 定期用新資料 fine-tune，對抗模型漂移
  • 容錯機制 — 模型失敗時 fallback 到規則或人工
  • 動態資源調配 — 自動擴容 / 限流
  • 版本控制流程 — Model Registry、A/B 測試、Canary Deployment
  • 監控 Dashboard — 即時追蹤延遲、錯誤率、漂移指標
一句話：「上線只是開始」 — 維運比訓練更難，模型不是靜態產品而是需要持續照顧的活物。

  • 有害內容 — 客服在敏感話題（自殺、種族）產出冒犯回應
  • 高風險領域誤導 — 醫療、法律、金融給錯誤建議，使用者照做出問題

真實案例：2023 紐約律師用 ChatGPT 寫狀紙，模型編造了 6 個不存在的判例 → 法官查證後重罰律師 5,000 美元 + 公開警告 — 成為 AI 幻覺最知名的法律事件。

處理機制
  • RAG — 把答案錨定在真實文件（連結 F 節）
  • 事實檢查機制 — 自動比對外部知識庫
  • 引用機制 — 要求模型給出來源，使用者可查證
  • 高風險領域（醫療、法律）必須人工審核
  • 產品上加風險預告聲明（"AI 生成內容僅供參考"）
  • 降低 Temperature — 減少模型亂創造的傾向
一句話：LLM 不會說「我不知道」，會自信地編 — 幻覺是結構性問題，不是 bug。
三層風險的「責任分工」與「處理時機」
① 資料層（資料工程團隊）事前預防 — 此時介入最便宜，1 元成本可省 100 元
② 模型層（ML / DS 團隊）事中緩解 — 必須有量化指標，否則沒有修正方向
③ 應用層（產品 / SRE 團隊）事後補救 — 此時介入最貴，可能伴隨品牌與法律代價
實務啟示：負責任的 NLP 不是某個職位的事，而是整條 pipeline 每個團隊都要負責任地交接。
常考點：三層風險的分類與配對 題目常問「下列情境屬於哪一層風險」或「下列何者為○層風險的對應處理機制」：
  • 資料層關鍵字：個資、匿名化、去識別化、版權、爬蟲、標註一致性、Data Bias
  • 模型層關鍵字：Model Bias、WEAT、公平性、詞向量偏見、性別／族群、Debiasing
  • 應用層關鍵字：諷刺／反語／方言、模型幻覺（Hallucination）、版本控制、即時推論、人工審核、RAG
  • 易混淆：「資料偏見（Data Bias）屬資料層」 vs 「模型偏見（Model Bias）屬模型層」 — 一個在源頭、一個在訓練過程
  • 必懂辨析：去識別化 = 可還原（代碼替換、抽象化、遮罩）；匿名化 = 不可還原（徹底刪除、隨機化、聚合）
  • 跨章連結：NLP 風險是更宏觀的 §4.3 AI 風險管理 的具體案例

電腦視覺（CV）技術與應用

電腦視覺（Computer Vision, CV）讓機器具備「看懂世界」的能力 — 從影像或影片中辨識、分析並理解環境資訊，支援人機互動、決策判斷與自動化。早期依賴人工設計特徵，2012 年 AlexNet 在 ImageNet 競賽的突破讓深度學習主導 CV。如今，CV 已深入自動駕駛、安防監控、醫療影像、智慧製造、零售服務。

D · 影像分類 — CNN 架構與知名模型

CNN 的基本架構

層功能備註
卷積層Convolutional用卷積核提取影像的局部特徵，生成特徵圖（Feature Map）核心，自動學習邊緣、紋理、形狀
池化層Pooling對特徵圖降維（最大池化 Max Pooling 最常用）減少計算量、增強泛化
激活層Activation引入非線性（ReLU 函數）提升表達能力
全連接層Fully Connected將特徵整合為一維向量進行最終分類分類輸出前的整合層
輸出層Softmax 函數轉為類別機率輸出每類別的可能性

知名 CNN 模型（按時間順序）

①
LeNet · Yann LeCun, 1998

早期 CNN 鼻祖

結構簡單（5 層），早期用於手寫數字辨識（MNIST）。為後續 CNN 奠定基本架構。

②
AlexNet · 2012

深度學習里程碑

在 ImageNet 競賽中脫穎而出，8 層結構引入 ReLU 激活 與 Dropout 正則化，奠定 CNN 在 CV 的地位。標誌深度學習時代正式來臨。

③
VGG · 2014

深度堆疊架構

強調深度與小卷積核（3x3），層數可達 19 層，分類準確度顯著提升。架構乾淨，至今仍是教科書範例。

④
ResNet · 2015

殘差連接突破

引入殘差結構（Residual Connection），允許網路深度達 152 層，解決深層網路的梯度消失問題。是後續所有超深網路的基礎。

影像分類常用評估指標

指標定義
準確率 Accuracy預測正確的樣本 / 總樣本數
Top-5 錯誤率真實類別位於模型預測的前五名內即視為正確；常用於類別眾多的資料集（ImageNet 1000 類）

E · 物件偵測 — 同時定位與分類

模型核心機制特性
YOLO
You Only Look Once		基於迴歸的單階段方法，同時預測邊界框與類別速度極快，適合即時應用（即時影片、自駕）
Faster R-CNN區域提議網路（RPN）生成候選框 → 分類 + 迴歸準確度高，但速度較慢；適合離線分析
SSD
Single Shot Detector		多尺度特徵進行物件檢測速度與準確度間平衡
選擇邏輯：速度 vs 準確度 即時應用（自駕車、監控、人流即時統計）→ YOLO；非即時但要求高精度（醫療影像、瑕疵離線檢測）→ Faster R-CNN；折衷場景 → SSD。實務上 YOLO 已迭代到 YOLOv8/v11，速度與精度同時持續提升。

F · 影像分割 — 像素級任務

類型定義區分點
語意分割
Semantic Segmentation		每個像素歸類為某預定類別（道路、建築、行人）不區分同類中的不同實例（所有車輛標為「車輛」）
實例分割

三大常見分割模型

FCN — Fully Convolutional Network

FCN

把傳統 CNN 應用於整張影像，透過上採樣層（Upsampling Layer）把低解析度特徵圖還原為原始影像大小，實現像素級分類。

優勢：端到端訓練、靈活。

U-Net

Medical Imaging Standard

專為醫療影像分割設計，特別適合處理不平衡資料集。對稱的編碼器-解碼器架構：Encoder 提取特徵、Decoder 恢復細節。

最大特點：跳躍連接（Skip Connections）讓解碼器能直接從高解析度特徵中恢復細節，大幅提升分割精度。

Mask R-CNN

Instance Segmentation

基於 Faster R-CNN 的擴展，結合物件偵測與像素級分割。引入額外的分支預測每個物體的二進制遮罩（mask）。

不僅能檢測物體，還能準確標註其邊界，對複雜背景情境尤其有效。

H · CV 三大風險與挑戰

RISK 01
資料隱私與合規
影像常含 PII（人臉、車牌、地址、兒童）。需匿名化（臉部遮蔽、模糊）、合法授權、保存期限限制、依 GDPR / 個資法管控。學校門禁人臉辨識若未獲家長同意可能引發倫理爭議。
RISK 02
模型偏見與可靠度
資料來源不平衡（膚色、性別、年齡集中）→ 對其他族群辨識準確度差。
緩解：偏見診斷（Bias Audit）、數據重加權、多維度評估（AUC、精確、召回）。
RISK 03
部署與維運
邊緣設備 GPU 受限 → 用輕量模型（MobileNet、YOLOv5-Nano）。
資料漂移影響：光照、攝影角度變化致模型失準。導入 MLOps + CI/CD、影像品質監測、預設容錯回退機制。
誤檢 vs 漏檢的不對稱成本 False Positive（誤檢）：把健康人誤判病人 → 不必要的進一步檢查與心理負擔。
False Negative（漏檢）：把瑕疵品判為合格 → 缺陷產品流入市場、可能造成大損害。
不同領域成本不同 — 醫療診斷通常要極低 FN（寧可誤檢也別漏抓）；垃圾郵件偵測則寧可漏抓也別誤刪正常信。設計時需依領域調整門檻。

生成式 AI 技術與應用

2022 年 ChatGPT 推出以來，生成式 AI（Generative AI）成為近年最受矚目的 AI 領域。從文字、圖像、影音、語音到程式碼，生成式 AI 不僅理解資訊，還能創造新內容 — 代表 AI 從「資料分析工具」進化為「具備主動創造價值的能力」。

補充：NVIDIA 的 AI 四階段演進說 2025 CES 黃仁勳指出 AI 經歷四階段：
① 感知 AI（Perception）理解影像、語音、文字。
② 生成式 AI（Generative）主動產生多模態內容。
③ 代理式 AI（Agentic）具備推理與計畫能力，自動完成複雜任務。
④ 物理 AI（Physical）能在真實世界移動、操控、互動（機器人、自駕車）。

五階段技術細節

① VAE 與 RNN 早期階段

2013-2015

變分自編碼器（VAE）透過學習資料的潛在分佈，從隨機噪聲中重建圖像或文本；強調機率建模。RNN 則專注序列資料生成。

代表應用：簡單圖像重建（MNIST 手寫數字）、基礎文本生成。

限制：生成結果模糊（低解析度）、品質不穩定、梯度消失難收斂。

② GAN 對抗生成網路

2014-2020 · Ian Goodfellow

核心機制：生成器（Generator）vs 判別器（Discriminator）對抗訓練 — 生成器學逼近真實資料分佈、判別器區分真假，形成動態平衡。

代表模型：DCGAN（深度卷積 GAN）、CycleGAN（風格轉換）、StyleGAN（高品質人臉生成 — 你看到的所有「不存在的人臉」AI 圖多半出自此）。

應用：人臉合成、圖像風格遷移（油畫 → 照片）、動畫角色生成。

限制：訓練不穩定（模式崩潰 Mode Collapse）、對超參數敏感、長序列複雜內容難。

③ Transformer 崛起

2017+ · Google

自注意力機制（Self-Attention）克服 RNN 在長距離依賴的限制，並行計算大幅提升；催生大型語言模型（LLM）。

代表模型：BERT（雙向理解）、GPT 系列（自迴歸生成）、T5（文本到文本）。

影響：生成內容的連貫性與邏輯性大幅提升，為後續多模態生成奠定基礎。

④ 擴散模型與多模態

2020+

擴散模型（Diffusion Models）透過逐步去噪（Denoising）過程從隨機噪聲生成細節豐富的圖像；代表如 DDPM、Stable Diffusion。

多模態模型整合圖像、文本、語音 — 例如 CLIP 把圖文對應到同一語意空間。

代表：Stable Diffusion（圖像）、DALL·E（文字到圖）、Flamingo（視覺問答）、Gemini（多模態整合）。

優勢：更高解析度、細節更豐富、處理多模態任務。

⑤ 巨型模型與代理式 AI

2023+

百億至千億參數的巨型模型（GPT-4、Claude 2、Gemini 1.5）不僅生成能力強，還展現自主性與工具整合能力。

技術特徵：巨型參數、指令驅動（Instruction Tuning）、工具鏈整合（API、RAG、記憶功能）。

趨勢：從「單純創造」轉向「任務解決與智慧協助」 — Agentic AI。

G · 生成式 AI 的五大潛在風險

RISK 01
內容真偽與可信度
生成模型可創造看似真實但虛構的內容 — 偽造新聞、改造人物影像、合成語音。可能誤導選舉、金融、市場、個人名譽。
RISK 02
智慧財產權與授權
訓練資料可能含受著作權保護的內容；未經授權即衍生爭議（NYT 控告 OpenAI）。歐盟 AI Act 要求大型基礎模型公開訓練資料摘要。
RISK 03
偏見放大與倫理
訓練資料的性別、種族、文化偏見會被模型無意延續甚至強化；對醫療、司法、招聘有倫理風險。
RISK 04
深偽（Deepfake）濫用
圖像 + 語音合成 → 深偽影片 → 詐騙與假訊息散播工具。需偵測技術 + 法律規範共同因應。
RISK 05
可控性與不確定性
結果不完全可預測 — Prompt 不清時可能生成不相關或不適當內容；對商業應用與決策系統的可靠性影響大。

多模態人工智慧應用

多模態 AI（Multimodal AI）的核心能力是同時處理並整合來自不同模態的資料 — 文字、影像、語音、感測訊號。相較於只處理單一型態的模型（NLP 只處理文字、CV 只處理影像），多模態強調跨模態訊息的理解、融合與應用。

為什麼多模態 AI 是必然趨勢 真實世界的決策從來不是「只看一種訊號」 — 智慧客服需同時分析語音、語意、表情；醫療診斷需結合病歷、影像、生命徵象；自動駕駛需融合攝影機、雷達、LiDAR。多模態 AI 把這些孤立的能力連起來，在複雜情境中展現更高彈性與準確性。

C · CLIP — 圖文對齊與對比學習

CLIP（Contrastive Language-Image Pre-training）由 OpenAI 於 2021 年提出，是多模態 AI 的開創性模型。用對比學習實現圖文語意對齊 — 訓練時把語意相符的圖文對在向量空間拉近、不符的推遠。訓練資料量達4 億對圖文。

CLIP 訓練示意：圖文嵌入到同一語意空間

CLIP 三大功能與應用

功能應用
圖文匹配Image-Text Matching判斷圖片與文字描述是否相符（圖 + 「貓在草地」→ 匹配度）
圖文檢索Image-Text Retrieval文字搜尋圖片（用「紅色跑車」搜圖庫）或圖片反向搜尋描述
圖片標題生成Image Captioning根據圖片生成語意描述（風景照 → 「夕陽下的海灘」）
補充：CLIP 為什麼是後續生成模型的基礎 DALL·E、Stable Diffusion 都依賴 CLIP 提供的「圖文對應關係」 — 訓練時可用 CLIP 判斷生成圖是否符合文字描述，作為損失函數的一部分。沒有 CLIP，就沒有「打字生成圖」的時代。零樣本學習能力強 是 CLIP 最大優勢：未針對特定任務訓練即可分類（如把貓的圖丟給 CLIP 配對到「a photo of a cat」即可得到分類）。

D · 語音模態的整合應用流程

1
STAGE 01 · ASR

語音辨識（Automatic Speech Recognition）

把語音訊號轉為文字。代表模型：Whisper（OpenAI）。

2
STAGE 02 · NLU

自然語言理解（Natural Language Understanding）

解析文字內容，提取意圖或語意。代表模型：BERT、GPT。

3
STAGE 03 · ACT + TTS

任務執行與語音回應

根據理解結果觸發動作或生成語音回饋。TTS 把文字回應轉為自然語音。

實例：「今天天氣如何？」的完整流程 語音輸入 → ASR 轉文字「今天天氣如何」 → NLU 解析意圖「查詢天氣 + 今天 + 當地」 → 呼叫天氣 API → NLG 生成回應「今天晴朗，氣溫 25 度」 → TTS 轉語音輸出。整個流程涉及語音 / 文字 / API / 語音四個模態的協同。
應用情境說明
語音助理智慧音箱（Alexa、Google Home）— ASR + NLU 實現語音指令控制
語音搜尋支援語音輸入查詢，提升搜尋便利性
客服系統自動化語音回應，提升服務效率

多模態 AI 不只整合語言與視覺，還可整合感測器資料（雷達、超音波、溫濕度、GPS、加速度計），提升環境感知能力。

領域融合方式效益

AI 導入評估

企業與組織已經從「是否導入 AI」進入「如何有效導入 AI」的轉折點。AI 的價值不只在技術先進，更在能否與業務目標、策略需求緊密結合，創造實際商業效益。本節建立從需求、技術、財務、風險、組織五面向的導入評估框架，適用於不同規模與產業。

AI 導入評估要回答五個關鍵問題 ① 我們需不需要 AI？
② AI 導入是否值得投入資源？
③ 我們要如何導入 AI？
④ 我們是否準備好導入 AI？
⑤ 有哪些潛在風險？

A · 業務需求與痛點分析

第一步是辨識組織內是否存在可由 AI 解決的具體問題。透過跨部門訪談、利益關係人訪談（Stakeholders）、流程觀察與資料分析，勾勒「現況流程（As-Is）」與「理想流程（To-Be）」之間的差距。

核心問題具體判斷標準
哪些業務流程可作為 AI 導入目標？重複性高、決策規則清晰、數據驅動的流程 — AI 介入效率最高
是否存在流程瓶頸或服務缺口？人力不足或效率問題的環節 — AI 可輔助或替代
有需要提升的關鍵 KPI？效率、準確率、回應速度 — AI 導入應對齊具體業務目標
避免「追隨技術潮流而盲目導入」 若沒有明確的業務痛點，僅因為「同業在做」或「老闆說要做」而導入 AI，極易導致資源浪費、效果不佳、員工不買單。需求的真實性與優先順序，是 AI 評估的第一道關卡。

B · 技術可行性評估

FACTOR 01
資料品質 Data Quality
AI 模型效能高度依賴資料的完整性、一致性、代表性。需評估是否具備高品質結構化資料、是否已完成標註。
FACTOR 02
IT 能力與系統整合
是否擁有足夠基礎設施支撐訓練與部署 — 伺服器、頻寬、雲端架構、資料倉儲。
FACTOR 03
AI 適配性
業務情境適合監督式 / 非監督式 / 強化式 / 生成式哪一種？深度學習 vs 統計模型的選擇。
條件不足？分階段推動 若技術尚未成熟，可納入中長期規劃：
① 先建立健全的資料治理架構。
② 加強 IT 基礎設施（雲端資源、網路帶寬）。
③ 選擇較簡單的業務情境做小範圍試點。
逐步積累實戰經驗，最終實現大規模落地。

C · 成本與效益分析

成本組成

  • 內部人力資源（資料工程師、AI 工程師、PM）
  • 技術訓練（員工 AI 素養提升、特定工具培訓）
  • 資料清洗與標註
  • 模型開發與調校
  • 系統整合（與既有 ERP / CRM 串接）
  • 第三方顧問或平台授權

三大財務評估指標

投資報酬率 ROI

Return on Investment

意義：衡量投資效益 — 每投入一元帶來多少回報。

ROI = （投資回報 − 投資成本）/ 投資成本 × 100%

應用：評估 AI 專案是否具足夠經濟回報。需考量導入後的營運成本、效益提升、時間節省。

淨現值 NPV

Net Present Value

意義：未來現金流量的折現總和，減去初期投資成本。

NPV = Σ（每期現金流 / (1 + 折現率)期數）− 初期投資

判讀：NPV > 0 表示專案能帶來超過初期投入的價值，值得做。NPV 考量「時間價值」，比 ROI 更嚴謹。

回收期 Payback Period

Payback Period

意義：投資成本完全回收所需的時間。

回收期 = 初期投資成本 / 每年淨現金流

判讀：回收期越短，投資風險越低，企業更快獲利。侷限：未考量時間價值，需與 NPV 搭配使用。

D · 組織成熟度盤點

即使 AI 技術可行，若組織數位成熟度、資料治理能力、跨部門協作機制不足，仍會推動困難。

① 數位與資料成熟度

  • 數位成熟度（Digital Maturity）：現行流程是否已數位化？ERP / CRM / 資料倉儲是否建置完善？
  • 資料治理（Data Governance）：資料的正確性、可追溯性、權限控管是否完備？是否有資料標準與管理單位？
  • 資料取得流程：不同系統資料是否可整合？是否具備 API、ETL 或自動化資料流？

② 團隊能力與跨部門協作

  • 組織能力（Organizational Capability）：是否有 AI、資料分析、系統整合的核心人才？無則須培訓或外部合作
  • 跨部門協作：是否有統一窗口與專案負責人整合需求與進度？
  • 導入推動責任單位：有無 AI 專案辦公室、數位轉型小組？

③ 資源與預算支持

  • 資源盤點：預留的資金、人力、時間是否充足？能否靈活調度？
  • 預算與來源規劃：是否為年度編列？有無補助計畫、投資評估、內部專案提案機制？
  • 導入條件：可提供試辦環境、測試資料或專責團隊嗎？

E · 風險識別與初步應對

風險類別典型問題
法規遵循與隱私個資法、GDPR；資料合法來源；資料跨境傳輸與外包；侵權與版權（生成式 AI 創作內容）
技術風險與穩定性資料缺口、模型穩定性、黑箱風險（缺乏可解釋性）、生成內容不確定性
組織文化與變革阻力員工抗拒（擔心被取代）、決策文化（經驗 vs 數據）、跨部門配合
補充：可解釋 AI（XAI）的應用情境 深度學習模型因高度非線性難以解釋預測依據，引發信任問題。XAI（Explainable AI）提供工具讓使用者理解模型決策依據。
關鍵應用：
① 醫療診斷（醫師需理解模型判斷依據）。
② 金融風控（向主管機關說明風險評分組成）。
③ 招聘 / 教育（避免歷史偏見導致歧視）。
④ 公部門 / 法務（決策需法律審查）。
常見技術：LIME、SHAP、Anchors、CounterFactual Explanations、Attention Visualization。

F · 概念驗證（POC）— 評估的最後一站

完成前面所有評估後，邁向實際導入前應規劃一個具體且可控的 POC（Proof of Concept）。POC 不是模型開發，是「先用最小範圍驗證能不能做」的實證 — 為後續導入規劃與風險管理建立實證基礎。

POC 條件設計：可控且具代表性

1
CRITERIA 01

範圍清晰、風險可控

以小範圍單位、部門或資料集進行測試，避免影響整體營運。測試出問題時容易解決、減少對業務的影響。

2
CRITERIA 02

具代表性業務流程

選擇能代表未來擴展應用邏輯的場景；該場景具實用與延展價值，能反映後續全局導入的挑戰與機會。

3
CRITERIA 03

資料與技術資源具備初步條件

數據格式、存取權限與基礎 IT 支援已準備就緒，避免因資源不足造成測試中斷。

4
CRITERIA 04

利益關係人支持度高

選擇有支持共識與合作意願的業務單位進行測試，提高推動成功的機率與快速反饋的可能。

POC 評估指標

指標類型具體內容用途
量化指標（效能）準確率、處理速度、錯誤率、成本節省與既有 KPI 比較，觀察 AI 導入後是否顯著優化
質性指標（使用者）問卷、訪談、使用日誌關注使用障礙、文化抗拒、流程磨合問題
技術適配性能否與現有系統整合？資料架構兼容否？影響後續部署可行性
組織接受度部門或使用者的正向接受與學習意願對後續推動至關重要
常考點：商業價值評估 vs 技術評估 題目給情境問「下列何者屬於商業價值評估的考量？」 — 答：預期效益與回報（如 ROI、節省成本）。技術評估則聚焦技術成熟度與部署資源。記住：商業評估看「值不值得」、技術評估看「做不做得到」、組織評估看「準不準備好」、風險評估看「會出什麼狀況」。

AI 導入規劃

完成評估後，要把「可行性與價值潛力」轉化為「具體可執行的規劃行動」。這個階段承上啟下 — 質量直接影響後續技術部署、專案達成、效益實現。本節介紹規劃步驟與專案管理工具。

沒有一體適用的導入模式 不同產業與規模在資料基礎、技術能力、治理文化、預算彈性上差異顯著：製造業重視感測器資料整合與即時性；金融業聚焦資料安全與法規遵循。同一企業內部各業務單位的優先順序也不同。導入規劃需具備高度彈性，根據組織實際情境調整。

A · 導入規劃的三大原則

PRINCIPLE 01
情境 / 目的導向
根據產業特性與業務需求定義專案目標與範疇 — 零售業聚焦銷售預測，醫療業強調診斷輔助。
PRINCIPLE 02
資源盤點
評估資料、技術人力、預算，確保專案規模與資源匹配。資料品質不足時應先投入資料整備而非急於模型開發。
PRINCIPLE 03
彈性調整
因應需求改變或技術瓶頸動態調整時程與策略。導入規劃的重點不在照表操課，而是靈活應對實際狀況。

C · 任務分解與時程設計

① 甘特圖（Gantt Chart）

以時間為橫軸、任務列表為縱軸的視覺化工具，呈現各項工作的時間安排、重疊區段、關鍵路徑（Critical Path）。AI 專案的甘特圖通常包含：

  • 問題定義與需求訪談（1-2 週）
  • 資料準備與標註（數週至數月）
  • 模型設計與訓練（依複雜度而定）
  • 驗證測試與反饋調整（多次迭代）
  • 部署與落地

② 工作分解結構（WBS）

採由上而下的拆解方式，把複雜任務分割為可控的子任務。例如「資料準備」可拆解為：

  • 資料源盤點 — 確認可用資料來源與格式
  • 資料清理 — 移除缺失值與異常值
  • 欄位標準化 — 統一資料格式與單位
  • 欄位對齊與標註流程建立 — 確保多源資料一致並完成標註

每個子任務需定義責任角色、預期成果、所需資源、預估時長。

③ 敏捷開發（Agile）— Sprint 短衝迭代

面對需求快速變動與技術高度不確定的 AI 專案，傳統瀑布式開發難以應對。許多 AI 團隊採用 Scrum 敏捷開發框架。

Scrum 框架的三角色

3 ROLES

Product Owner（產品負責人）：定義產品需求與優先順序，維護 Product Backlog，確保開發內容符合業務價值。

Scrum Master（敏捷教練）：協助團隊遵守 Scrum 流程、移除障礙、促進團隊合作與持續改善。

Development Team（開發團隊）：跨職能成員，負責實際完成 Sprint 內任務（資料處理、建模、測試、整合）。

五個關鍵活動

5 EVENTS

Sprint Planning：決定本次 Sprint 要完成哪些任務。

Daily Scrum（每日站會）：15 分鐘同步團隊進度與阻礙。

Sprint（短衝）：固定週期的開發節奏，通常 1-4 週（常見 2 週）。

Sprint Review（展示）：展示 Sprint 成果，接受回饋。

Sprint Retrospective（回顧）：檢討過程，提出改善措施。

補充：Sprint 在 AI 專案中的常見目標 「建立初版資料處理流程」「完成模型訓練並達 85% 準確率」「部署原型 API 並開放內部測試」… 每輪 Sprint 結束後執行展示 + 回顧 + 下輪規劃三個活動，讓專案能快速驗證、修正、強化，有效降低後期失敗風險。

D · 跨部門協作機制

  • 協作流程設計：明確各部門責任與任務範圍（資料提供、模型驗收、系統整合），避免重疊與漏洞
  • 例行同步節奏：定期專案會議與進度報告（常見每週例會）
  • 共用工具平台：任務看板（Jira / Trello）、文件平台（Notion / Confluence）、即時通訊（Slack / Teams）

E · 三種 AI 導入模式

① 選用現有商業服務（Low-code / No-code）

SaaS · AutoML

適合：技術能力有限或時間緊迫的企業；快速建置原型。

優勢：快速部署、技術門檻低、靈活。

挑戰：① 彈性有限（AutoML 難調整深層結構）；② 長期訂閱成本累積；③ 資料隱私風險（雲端處理）。

代表：Google AutoML、Azure ML Studio、Hugging Face Hub、ChatGPT API。

② 內部技術團隊開發

In-House

適合：具備技術人力與硬體資源、有長期 AI 發展計畫的企業。

優勢：掌控力強、適應性高，可深度客製化並保留技術自主性。

挑戰：需大量時間與資源（招聘、設備、流程建立），初期成本較高。

適用案例：金融機構詐欺偵測（內部資料 + 合規）；科技公司影像辨識深度優化。

③ 委外開發

Outsourcing

適合：缺乏內部技術能力但有預算支持的企業。

優勢：專業性與執行效率高、能縮短開發時程。

挑戰：① 合約管理（交付範圍、時程、品質）；② 資料安全風險（敏感資料外流）；③ 後續維運轉移成本。

建議：明確合約條款（交付物、SLA、責任歸屬），保留關鍵知識與技術的內部累積。

F · POC 驗證與規劃成果整合

完成導入規劃後，最後一步是用 POC 驗證技術與流程可行性，並系統化整理規劃成果。

POC 的小範圍快速迭代

階段關鍵動作

常考點：人員規劃與跨部門協作 題目給情境問「AI 專案初期規劃階段，下列哪一項屬於人員規劃的重點？」 — 答：AI 專案所需的跨部門協作角色（不是 API 規格、不是模型參數、不是作業系統版本）。AI 專案涉及資料、業務、IT 等多部門，規劃階段就需明確分工與責任分配。

AI 風險管理

AI 風險不只來自技術落差或部署複雜性，更牽涉資料偏誤、模型不穩定、演算法不透明、個資濫用、法規責任、社會倫理等多重層面。若缺乏前期辨識與持續治理，可能造成導入失敗、企業聲譽損害、法規合規重大威脅。

AI 風險管理 = 全生命周期治理 不應只是部署後的輔助工作，而要從規劃階段就納入，貫穿模型開發、測試、部署、運維各階段，形成完整的全生命周期風險治理體系。

A · 三大風險類型

① 技術層風險

資料品質風險

Data Quality Risk

AI 模型對資料高度依賴；資料缺失、錯誤、不完整或不具代表性 → 模型不穩定、預測誤導，甚至嚴重後果（醫療誤診、金融失準）。

常見問題影響
資料遺漏（Missing Data）輸入錯誤、資料丟失、未收集 — 影響訓練與預測準確
資料偏頗（Data Bias）樣本分佈不均 — 對特定群體預測不公平
不正確標註（Incorrect Labeling）人工或機器標錯 — 直接影響學習結果
資料漂移（Data Drift）輸入資料分佈隨時間改變 — 模型逐漸偏離原有準確度

影響案例：信用評分系統因歷史資料偏差導致特定族群被歧視。

模型穩定性風險

Model Stability Risk

實際應用中，模型可能因輸入變動或外部環境改變而預測準確度下降甚至失效。

類型說明
模型漂移（Model Drift）預測準確度隨時間變差 — 訓練資料或環境變化
過擬合（Overfitting）在訓練集表現好但對新資料不行 — 死背了訓練資料
概念漂移（Concept Drift）資料背後的潛在關係隨時間改變 — 常見於長期運行模型

影響案例：製造業瑕疵辨識系統在生產線微小調整後精準度下降。

補充：Data Drift vs Concept Drift Data Drift（資料漂移）：X 的分佈變了 — 例如使用者人口結構改變、新使用者介入。
Concept Drift（概念漂移）：X 與 y 的關係變了 — 例如詐騙手法升級、購買行為因疫情變化。
兩者都需要持續監控，但解法不同：Data Drift 可能用重新校準解決；Concept Drift 通常需要重新訓練模型。

風險說明

Cross-functional		業務、技術、法務部門協作鬆散 → 需求落差、資源浪費、延誤交付。

例：標註者對特定群體有刻板印象
BIAS 04
演算法偏見
演算法優化整體效能 → 犧牲特定群體公平性。
例：推薦系統忽略少數群體需求

E · 風險監控與 MLOps

MLOps（Machine Learning Operations）把 ML 開發與營運整合，自動化模型建置、部署、監控、修正全流程，有效管理模型全生命周期。

MONITOR 01
模型漂移監控
追蹤模型效能下降，找原因（資料變化、環境因素）。
例：電商推薦模型準確率下降時自動觸發重訓
MONITOR 02
資料漂移監控
監測資料分佈變化，輸入與訓練分佈差異大時發警告。
例：節日促銷期間特徵分佈變化的即時調整
MONITOR 03
效能異常警示
錯誤率過高時即時警報（預警閾值如 5%）。
例：金融詐欺偵測模型準確率下降時即時通知
MONITOR 04
自動重新訓練
監控發現效能下降或漂移 → 自動觸發重訓。
例：電商推薦模型自動重訓恢復 90% 準確率
常考點：風險類型與應對策略對應 題目給情境問「下列何者屬於 AI 模型偏誤造成的風險？」 → 答特定族群預測結果不公平。問「長期輸入資料分佈改變導致效能下降，最適合的風險管理作法」 → 答建置概念漂移監控與觸發模型再訓練流程。問「決策透明性不足是什麼風險」 → 答黑箱風險與可解釋性（XAI 是緩解策略）。

重複值 Duplicates針對主鍵（顧客編號、交易編號）做重複檢查；用 pandas.drop_duplicates() 或 Spark DataFrame 方法處理
異常值 Outliers統計方法（Z-score、IQR）或機器學習方法（Isolation Forest、LOF）

QUALITY 02

QUALITY 03

QUALITY 05

C · 特徵工程

把原始資料轉換為適合機器學習模型學習的格式與結構，提升模型表達力與預測效能。

① 特徵處理流程

類型方法用途
數值正規化
Normalization		Min-Max（縮放至 0-1）避免單位差異造成模型學習偏誤
數值標準化
Standardization		Z-score（平均 0、標準差 1）梯度模型（線性迴歸、SVM、神經網路）必用
類別 — 獨熱編碼
One-hot Encoding		每類別轉為二進位欄位無序類別變數（城市、顏色）
類別 — 標籤編碼
Label Encoding		類別轉為整數有序類別（高中、學士、碩士）
時間戳特徵擷取週期性（星期幾、上午/下午、工作日/週末）捕捉時間與週期性模式
文字處理TF-IDF、Word Embeddings文字轉數值供模型處理

② 特徵選擇與降維

方法原理
資訊增益
Information Gain		衡量特徵在預測中帶來的資訊量
皮爾森相關係數衡量數值特徵之間的線性相關性
L1 正則化（Lasso）懲罰模型複雜度來選擇最具影響力的特徵；常用於線性模型
PCA 主成分分析找資料中方差最大的方向降維；高維資料 + 計算負擔大
t-SNE / UMAP常用於視覺化資料分佈，便於觀察特徵空間的分類趨勢

③ 自動特徵工程（Auto Feature Engineering）

AutoML 平台（Google AutoML、H2O.ai、AutoKeras、TPOT）提供自動化特徵工程功能：

  • 自動特徵創建：日期拆分（年、月、日、星期）、基於現有特徵的數學運算（加法、乘法、對數）、多項式特徵、時間序列滯後特徵
  • 自動篩選有用特徵：用隨機森林或 L1 正則化評估特徵貢獻，自動刪除冗餘
  • 特徵組合與互動作用檢測：測試各特徵組合對預測結果的影響
  • 適應不同資料集：類別資料自動選編碼方法、缺失值自動選填補方式
  • 基於模型反饋的特徵選擇：不斷訓練與驗證多個模型，智慧選擇最佳特徵集
補充：AutoML 的優勢與限制 優勢：
① 加速資料科學流程（從原始資料到模型訓練的時間大幅縮短）。
② 降低技術門檻（即使是非專業使用者也能完成）。
③ 高效能（快速評估大量組合）。
限制：對特定領域知識的整合仍有限、生成的特徵可能難以解釋、複雜資料情境下需人工干預。實務上 AutoML 是「省工不省腦」 — 仍需業務理解作為起點。

D · 模型選擇 — 六大類型

模型選擇是技術落地前最具策略性的決策。它影響預測準確率、效能表現、系統可維護性、擴展性、運營成本。

① 監督式學習 — 迴歸任務

預測連續數值型結果。應用：銷售量預測、房價預測、需求預測。

演算法特性與適用
線性迴歸
Linear Regression		最簡單；資料具線性關係時適用
決策樹迴歸樹狀結構；適用於非線性；解釋性強但易過度擬合
隨機森林迴歸多棵決策樹的平均；減少過度擬合
梯度提升樹
Gradient Boosting		集成學習；多次迭代提升準確度；複雜資料集表現良好

② 監督式學習 — 分類任務

將資料點分類到不同的離散類別。應用：垃圾郵件、疾病診斷、客戶流失預測、金融風險評估。

演算法特性與適用
邏輯迴歸
Logistic Regression		透過 Sigmoid 函數將預測值轉為機率；二元 / 多元分類；特徵與類別線性可分時最佳
決策樹分類條件分支；解釋性佳；非線性與混合型特徵；易過擬合
隨機森林分類多棵決策樹投票；高維 / 複雜結構資料表現好
SVM 支持向量機找最佳超平面分隔；高維資料、邊界清晰；核函數處理非線性
KNN K 最近鄰根據鄰近資料的類別預測；直觀；對量與維度敏感；小型資料集為主
梯度提升樹分類集成學習；逐步修正錯誤；多數分類問題穩定優異

③ 非監督式學習（Unsupervised）

資料無標註的情況下進行分析；從資料中找出內在結構、群集、分佈模式。常用於探索性分析、特徵降維、預處理階段。

演算法特性與適用
K-means劃分 K 個群集；簡單高效；須事先給定 K 值；對初始點敏感
DBSCAN基於密度的分群；發現任意形狀群集；自動標示離群點；不需指定群集數
階層式分群
Hierarchical Clustering		建立樹狀層級（Dendrogram）；無需指定群集數；探索分群層級結構
PCA 主成分分析降維方法；用於視覺化與特徵壓縮

應用：顧客分群（行為與紀錄）、異常偵測（詐欺、資安）、特徵壓縮、資料視覺化。

④ 深度學習（Deep Learning）

架構特性典型應用
CNN 卷積神經網路處理圖像 + 空間特徵；卷積、池化、激活人臉辨識、工業檢測、自動駕駛、醫療影像
RNN / LSTM / GRU時間序列資料；LSTM 解決長期依賴語音、文字、感測器；金融時序預測
Transformer自注意力機制；BERT、GPT 為代表NLP、語意理解、跨模態學習、智慧客服

⑤ 強化式學習（Reinforcement Learning, RL）

基於「試誤學習」的方法 — 智能體（Agent）在環境中嘗試行動、觀察結果、根據獎勵調整策略。無需明確標註資料，依靠互動經驗學習。

演算法特性
Q-learning基於值函數；學習每個狀態-行為對的預期回報；適用離散動作空間
DQN Deep Q Network結合深度學習與 Q-learning；處理高維輸入（如圖像）；Atari 遊戲突破
Policy Gradient直接學習狀態到行為的機率分佈；REINFORCE、Actor-Critic
PPO 近端策略優化OpenAI 提出；平衡更新幅度與學習效率；廣泛用於遊戲、機器人

應用：自駕車控制、智慧倉儲與物流、資源調度、遊戲 AI（AlphaGo、AlphaStar）。

⑥ 生成式模型（Generative Models）

演算法特性與應用
GAN 生成對抗網路生成器 vs 判別器對抗訓練；圖像生成、人臉合成、風格轉換；訓練不穩、易模式崩潰
VAE 變分自編碼器學習資料的潛在分佈；異常偵測、語音重建、隱含變數建模
Diffusion Models 擴散模型逐步加噪 → 反向去噪；DALL·E 2、Stable Diffusion；圖像品質明顯優勢；生成慢、運算成本高
常考點：給情境選模型
  • 顧客流失預測（會 / 不會）→ 分類任務（Logistic、決策樹）
  • 房價預測（連續數值）→ 迴歸任務（線性、隨機森林、Gradient Boosting）
  • 顧客分群（沒有標籤）→ 非監督式（K-means、DBSCAN）
  • 影像辨識 → CNN；時序預測 → LSTM / Transformer
  • 遊戲 AI / 自駕車控制 → 強化式學習（PPO、DQN）
  • 生成新圖像 → Diffusion / GAN
模型不是越複雜越好 選擇模型需綜合考量資料特性、開發時間、計算成本、後續維運難度、可解釋性。深度學習雖強，但需大量資料與算力；對於小資料、低延遲需求、需可解釋的場景，簡單模型（邏輯迴歸、決策樹）反而是更好的選擇。

AI 技術系統集成與部署

模型訓練完成只是起點。如果模型停留在實驗環境、無法部署至實際營運流程，就無法產生真正的商業價值。AI 部署不單是「模型上線」，而是涵蓋開發、測試、部署、監控、回訓、優化的完整技術流程。

AI 系統部署 vs 傳統 IT 部署 — 為什麼複雜？ ① 模型效能會隨輸入資料變化而衰退，必須具備彈性更新。
② 推論服務對延遲與可用性有更高要求。
③ 模型更新頻繁，需自動化重訓與版本控管。
④ 還需符合法規對資料使用與透明度的合規要求。

A · 系統集成的五大面向

① 資料來源與資料管線整合

Data Source & Pipeline

從多來源取得資料：企業內部資料倉儲（Data Warehouse）、資料湖（Data Lake）、IoT 即時感測資料、第三方 API。

建立完整的資料管線（Data Pipeline）：擷取 → 格式轉換 → 清理 → 入庫，確保資料能持續、穩定地流入模型運行流程。

② 企業內部系統整合

Internal Systems

模型預測結果整合回 ERP / CRM / POS / MES 等核心系統，讓 AI 服務真正支援業務端執行。

例：CRM 自動根據模型推薦寄送促銷信件；ERP 根據需求預測動態調整採購。

③ API 與平台接軌

API & Integration

模型推論結果需被前端（網站、App、BI 工具）即時取得 → 封裝為可存取的服務介面。

實作方式：RESTful API（最常見）；高效能或雙向通訊用 gRPC、Webhooks。需納入容錯、快取、負載管理機制。

④ 安全與權限控管

Security & Access Control

涉及個資、商業機密或金融應用時，安全特別關鍵：

  • 使用 OAuth2 或 JWT 進行身分驗證與授權
  • 不同角色設不同存取權限（部分用戶可重訓或調參、部分只能查詢）
  • 實作 API Gateway 進行存取日誌記錄與流量控管

⑤ 跨雲與異質系統整合

Multi-Cloud & Heterogeneous

實務上 AI 模型可能橫跨：本地資料中心（訓練）+ 邊緣裝置（推論）+ 雲端平台（如 AWS / Azure）。

為確保資料一致性與服務連續性，需設計中介整合層或服務中樞（如 Service Mesh），統一管理資料同步、服務調用、狀態追蹤，避免平台間的資訊孤島。

B · 系統架構設計

① 微服務架構（Microservices）

▲ 優點
  • 易於隔離錯誤（單一模組失效不影響全系統）
  • 適合跨團隊並行開發
  • 依模組需求彈性調整資源
▼ 挑戰
  • 模組間介接需設計標準 API
  • 分散式管理複雜度高

② 容器化與編排（Containerization）

工具功能
Docker容器映像建構與執行；確保跨平台部署一致性
Kubernetes (K8s)容器編排、資源調度、故障恢復；管理大規模容器集群

常見實作原則

  • 滾動更新（Rolling Update）：不中斷服務地更新容器版本
  • 自動擴縮（Autoscaling）：根據流量自動調整容器數量
  • 負載平衡（Load Balancing）：確保多個推論節點分擔請求壓力

③ 模型服務化與 API 封裝

把模型封裝為可透過 HTTP / REST 呼叫的推論服務 — Flask、FastAPI、TensorFlow Serving、TorchServe。

  • 支援多版本管理（Model Versioning）
  • 加入使用者驗證、速率限制
  • 提供推論結果的記錄與監控（Log & Metrics）

④ 可擴充與高可用性設計

  • 冗餘部署（Redundancy）：多節點部署避免單點故障
  • 容錯切換（Failover）：節點故障時自動切換至備援

D · MLOps — 模型運維與監控

MLOps（Machine Learning Operations）是 DevOps 在 AI 領域的延伸，把模型訓練、測試、部署、監控進行持續整合與自動化 — 確保模型在全生命週期中的一致性、效率與可追蹤性。

① 持續整合與持續部署（CI/CD）

階段內容
CI 持續整合原始碼與資料變動即觸發訓練與測試；整合單元測試、模型準確率驗證、效能基準比較
CD 持續部署通過測試門檻的模型自動部署至測試或生產環境；支援 A/B 測試或藍綠部署降低風險
模型版本控管MLflow、DVC 追蹤每次訓練的參數、資料版本、模型成果；建立可重現性
補充：MLflow 與 DVC MLflow：開源平台管理 ML 全流程 — 實驗記錄、模型訓練、部署、版本控制。
DVC（Data Version Control）：版本控制工具，專為 ML 設計，支援資料集與模型的版本追蹤。
傳統 Git 不擅長處理大檔案（資料集、模型權重），DVC 補上了這一塊。實務上常見 Git + DVC + MLflow 三件組。

② 模型監控與回饋（Monitoring & Feedback）

監控項目說明

資料漂移 Data Drift輸入資料的統計特徵改變
效能指標錯誤率、延遲、吞吐量

回饋與自動修正流程

  • 設定觸發門檻（如準確率下降 5%）
  • 啟動再訓練流程（Retraining Pipeline）
  • 透過熱更新（Hot Swap）或滾動更新替換模型版本
  • 整合 Log 分析與 Dashboard（Prometheus + Grafana）
  • 自動警告 / 提示系統，異常時通知工程人員

F · 跨部門整合機制

#答案解析
1(C)資料標註是監督式學習的關鍵步驟，能讓模型理解每筆資料對應的標籤或結果。
2(C)許多演算法（SVM、KNN）對特徵尺度敏感；標準化有助提升模型表現與穩定性。
3(A)顧客流失通常為「會 / 不會」的二元分類問題；適合 Logistic Regression、決策樹等。
4(B)選擇模型需考量資料特性、開發時間、計算成本與後續維運難度，不應一味追求複雜度。
5(C)插補法（Imputation）可用平均、中位、眾數或模型預測填補缺失欄位。
6(C)容器技術可封裝應用與依賴環境，確保模型從開發到部署保持一致。
7(C)MLOps 結合 DevOps 與機器學習，實現模型生命週期的持續整合、測試與監控。
8(B)邊緣 AI 將模型部署至手機、攝影機、感測器等裝置上，提升即時性並減少延遲。
9(C)模型部署後若輸入資料特性改變，將可能導致預測不準；故需建立監控機制。
10(C)產品經理負責需求對齊與驗證場景，工程人員負責將模型封裝並整合至應用流程。
